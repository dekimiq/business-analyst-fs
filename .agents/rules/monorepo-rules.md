# Корневые правила проекта `business-analyst-fs`

> Этот файл читается AI-агентом **перед любым действием** в репозитории.
> Правила обязательны. Исключения требуют явного подтверждения от разработчика в чате.

---

## 1. Общие принципы поведения агента

- **Перед написанием кода** — прочитай задачу полностью, определи какой модуль затрагивается, проверь существующие файлы в этом модуле.
- **Не создавай новые файлы** без необходимости. Если логика помещается в существующий файл — помести туда.
- **Не удаляй и не переименовывай** файлы без явной команды от разработчика.
- **Один шаг за раз.** Если задача крупная — предложи план и жди подтверждения перед реализацией.
- **Не трогай соседние модули** без необходимости. Задача в `apps/backend` — не правим `apps/bot-interaction`.
- **Не угадывай env-переменные.** Все переменные окружения берутся строго из `packages/env/src/index.ts`. Хардкод значений — запрещён.
- **Межмодульное взаимодействие** — только через HTTP API или BullMQ очереди. `apps/*` не импортируют друг друга напрямую.
- **Комментарии на русском языке.** Код (переменные, функции, классы) — на английском. Комментарии и JSDoc — на русском.
- **Бизнес-логика не живёт в контроллерах / хендлерах** — только в `services/`.
- **Секреты не логируются.** API ключи, токены, пароли — никогда в `console.log` / логгер.

## 2. Структура монорепо

```
./                             ← корень: .env, docker-compose.yml, turbo.json,
│                                 tsconfig.base.json, .prettierrc, .eslintrc
├── packages/
│   ├── env/                   ← ЕДИНСТВЕННЫЙ источник env-переменных для всех модулей
│   └── shared/                ← общие TypeScript типы и утилиты
├── apps/
│   ├── backend/               ← AdonisJS API, интеграции Yandex + AmoCRM, синхронизация
│   ├── bot-interaction/       ← Telegram + MAX бот, точка входа для клиента
│   ├── ai-module/             ← AI оркестратор + аналитик, RAG, tool calling
│   └── node-cron/             ← планировщик задач, выставляет задачи в BullMQ
└── infra/
    ├── nginx/
    └── postgres/              ← init.sql: схемы, пользователи, расширения
```

**Правила навигации:**

- Общие типы — только в `packages/shared/src/types/`, не внутри модуля.
- Утилиты используемые в 2+ модулях — только в `packages/shared/src/utils/`.
- Каждый `apps/*` самодостаточен и не импортирует другой `apps/*` напрямую.

## 4. Пакетный менеджер и команды

**npm** — основной пакетный менеджер проекта.

```bash
npm install                                        # установка зависимостей
npm install <pkg> --workspace=apps/backend         # добавить зависимость в модуль
npm install -D <pkg>                               # добавить devDep в корень

turbo dev                                          # запустить все модули
turbo build                                        # собрать все (с учётом зависимостей)
turbo type-check                                   # проверка типов
turbo lint                                         # линт всех модулей
```

## 5. Стилизация кода

Код best practices без лишних усложнений.

### Запрещено:

❌ any — запрещён, использовать unknown + сужение типа
❌ @ts-ignore — запрещён
❌ @ts-expect-error — только с обязательным комментарием-обоснованием рядом
❌ as-касты без предварительной валидации
❌ non-null assertion без обоснования

### Как нужно:

✅ Явные типы возвращаемых значений у функций
✅ unknown для внешних данных + zod для сужения
✅ Типы общих сущностей из @project/shared
✅ Ранний возврат вместо вложенных if

**Ключевые правила семантики кода:**

- **Точка с запятой `;` — запрещена.** Prettier удаляет её автоматически, агент не должен ставить.
- Строки — одинарные кавычки `'`, не двойные.
- Trailing comma везде (ES5+) — последний элемент массива/объекта получает запятую.
- Стрелочные функции всегда со скобками: `(x) => x`, не `x => x`.

Агент **не форматирует код вручную** — форматирование применяется через Prettier автоматически при коммите через Husky.

## 6. Переменные окружения

- Единый `.env` в корне, модульные `.env` внутри `apps/*` — **запрещены**.
- Доступ только через `@project/env`:

```typescript
// ✅
import { env } from '@project/env'
const host = env.DB_HOST

// ❌
const host = process.env.DB_HOST
const host = 'localhost'
```

- Добавление новой переменной: сначала в `.env` + `.env.example`, затем в схему `packages/env/src/index.ts`, затем использование.

## 7. Модульные правила

Этот документ описывает **глобальные и универсальные** правила. Перед работой каком-то из модулей агент **обязан прочитать документ с правилами модуля** в котором он работает.

Каждый модуль имеет собственный `[name]-rules.md` с деталями реализации:

- `apps/backend/.agents/rules/backend-rules.md` — AdonisJS, Lucid, структура контроллеров и сервисов
- `apps/bot-interaction/.agents/rules/interaction-rules.md` — GrammyJS, MAX, IBotAdapter паттерн
- `apps/ai-module/.agents/rules/ai-rules.md` — Vercel AI SDK, tool calling, RAG, оркестратор/аналитик
- `apps/node-cron/.agents/rules/cron-rules.md` — node-cron, BullMQ, расписание задач
- `packages/env/.agents/rules/env-rules.md` — добавление и изменение переменных окружения
